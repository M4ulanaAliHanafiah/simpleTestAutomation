# simpleTestAutomation

A - dari pengalaman saya, ada tiga penyebab utama yang bikin automation UI jadi flaky yaitu:
Hard wait di mana-mana. Kodenya biasanya penuh waitForTimeout(3000) sebelum setiap interaksi. Di lokal dengan koneksi bagus, memang cukup. Tapi di CI yang resource-nya terbatas atau lagi heavy load, 3 detik itu bisa tidak cukup. Hasilnya test gagal secara random, dan lama-lama orang mulai anggap itu normal. Padahal root cause-nya jelas dan bisa diperbaiki relatif cepat kalau mau.
Locator yang rapuh. Ini juga sangat umum. Biasanya locator-nya terikat ke struktur HTML atau CSS class yang sifatnya dinamis, kayak div.container > div:nth-child(2) > button atau class yang namanya di generate otomatis. Begitu developer refactor tampilan entah itu tambah wrapper div, ubah urutan elemen, atau ganti nama class untuk keperluan styling test langsung patah. Padahal fungsionalitasnya tidak rusak sama sekali. Ini yang bikin tim kehilangan kepercayaan ke automation, karena mereka tidak bisa bedain mana test gagal karena ada bug, mana yang gagal karena locator-nya yang bermasalah.
Test tidak independent. Yang ini paling susah di debug karena dependency nya tidak kelihatan secara eksplisit di kode. Contoh yang pernah saya temui: test untuk fitur Edit Profile bergantung pada session aktif yang ditinggalkan oleh test Login sebelumnya. Kalau test Login di-skip atau gagal duluan, test Edit Profile ikut gagal padahal Edit Profile-nya sendiri tidak ada masalah. Di CI yang kadang jalankan test secara paralel atau dengan urutan yang berbeda, situasi kayak gini langsung kelihatan dan hasilnya jadi tidak konsisten.

C - Kalau ada test yang gagal di CI tapi selalu lulus di lokal, langkah pertama yang saya lakukan adalah baca log CI sampai habis bukan cuma lihat error di line terakhir. Error message paling bawah itu sering menyesatkan. Misalnya tertulis TimeoutError: element not found, tapi kalau scroll ke atas ada request ke API yang return 500 duluan itu yang jadi root cause nya, bukan soal element nya.
- Setelah itu saya perhatikan di step keberapa test-nya gagal. Kalau gagal di awal sebelum ada interaksi apapun, kemungkinan masalahnya di environment halaman tidak bisa diakses atau terlalu lama load. Kalau gagalnya di tengah-tengah flow, lebih condong ke race condition atau elemen yang belum siap saat di interact.
- Langkah berikutnya saya coba reproduce di lokal tapi dalam kondisi yang lebih mirip CI yaitu jalankan Playwright di mode headless. Banyak bug yang hanya muncul di headless tapi tidak kelihatan di headed mode misalnya elemen yang ada tapi di luar viewport, atau animasi yang belum selesai. Kalau masih tidak bisa reproduce, saya lihat CI artifacts. Playwright bisa dikonfigurasi untuk simpan screenshot dan video saat test gagal, dan dari situ biasanya sudah langsung kelihatan apa yang terjadi.
- Untuk komunikasi ke developer, saya tidak langsung bilang ini bug. Saya share temuannya dulu dengan konteks yang jelas dan memberikan step dengan screenshot nya.
- Soal keputusan akhirnya perbaiki, disable, atau delete kriterianya buat saya cukup straightforward. Kalau root cause sudah ketemu dan fix-nya jelas, misalnya tinggal ganti hard wait jadi conditional wait, saya perbaiki langsung. Tidak perlu tunggu sprint berikutnya untuk hal yang bisa selesai dalam 30 menit. Kalau butuh investigasi lebih dalam dan tidak bisa selesai hari ini, saya disable tapi wajib ada komentar kenapa di disable dan ada tiket yang linked dengan deadline yang jelas. Test yang di-disable tanpa keterangan itu sama aja kayak utang yang tidak ada catatannya, tidak akan pernah dibayar. Dan kalau ternyata fiturnya sudah tidak ada atau sudah berubah total, test nya dihapus saja tidak ada gunanya mempertahankan test yang sudah tidak relevan, hanya nambah noise dan bikin orang makin tidak percaya sama hasil automation.
